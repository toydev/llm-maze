# ドキュメントガイド

## 前提

- 未来に向けて書く: あるべき状態を書く
- ソース自体に語らせる: ドキュメントよりソース修正を優先、ドキュメントによるつじつま合わせはしない

### エージェントの特性と対応
- **記憶リセット**
  - 特性：セッション間の記憶の引継ぎがない
  - 対応：即時理解のドキュメントを目指す、知識への到達性を考慮する
- **着目限界**
  - 特性：指示・ルールが多すぎると重要な指示が顧みられない
  - 対応：簡潔なドキュメントを目指す、自動リマインダーを活用する
- **自動リマインダー**
  - **~/.claude/CLAUDE.md**: システムプロンプト（常に有効）
  - **実行ディレクトリ/親ディレクトリのCLAUDE.md**: システムプロンプト（常に有効）
  - **その他ディレクトリのCLAUDE.md**: 該当ディレクトリ初回Read時に自動リマインド

## ドキュメント配置ガイド

**前提**: セッションはDQAIルートで開始

- 常時必要 → ~/.claude/CLAUDE.md・DQAI/CLAUDE.md
- モジュール作業時特化 → モジュール/CLAUDE.md（control-hub/CLAUDE.md、game-machine/CLAUDE.md）
- 特定作業時特化 → DQAI/docs/配下の専用ドキュメントファイル、モジュール/docs/配下の専用ドキュメントファイル
- 該当リソース参照時特化 → モジュール/サブディレクトリ/CLAUDE.md・個別ファイル内コメント

## ドキュメント内容ガイド

### 書くべきもの（Why重視）
- 背景・前提・制約: 根拠となる考え方・価値観
- 設計思想: なぜその実装にしたのか
- 課題解決: どんな問題をどう解決しようとしているか
- 判断基準・行動指針: 開発者が判断・行動するための具体的な指針
- 具体例: 抽象的な説明を補完する実例

### 書かざるべきもの
- 実装手順: コードを読めば分かる処理の流れ
- 変数の説明: 変数名から自明な内容
- 基本的なAPI使用法: 標準的なライブラリの使い方
- 一時的内容: 作業経過・デバッグ
- 経緯: 作業報告・経緯
- 抽象的な一般論: どのプロジェクトにも当てはまる美辞麗句
- 事実の列挙: ファイルを見れば分かる状態の記述
